# Lunaris

A local-first harness for running autonomous coding agents across many repositories at once.

Lunaris gives each repository its own isolated AI environment and a persistent orchestrator that breaks a goal into a task graph and works through it with specialized subagents. Everything an agent does is recorded to an append-only event log, which feeds live observability, cost analytics, and a self-tuning optimizer. A policy engine and capability tokens keep unattended runs inside their lane, and a Mission Control web app lets you watch and steer the whole thing.

It runs entirely on your machine. Storage is SQLite, all cryptography uses Node's built-in `node:crypto`, and there are no native dependencies to compile.

The full design rationale lives in [LUNARIS_SPEC.md](LUNARIS_SPEC.md).

## Highlights

- **Per-repository environments.** A committed `lunaris.toml` declares each project's models, providers, budgets, and policy. Secrets stay in a gitignored `.aienv`.
- **Orchestrator + subagents.** One persistent orchestrator per project plans and delegates to confined coder/reviewer/researcher subagents.
- **Multi-provider gateway.** Anthropic, OpenAI, DeepSeek, any OpenAI-compatible endpoint, and local Ollama, behind one streaming API with a transactional budget ledger.
- **Graph memory.** A per-project knowledge graph the agents treat as advisory context, with a retention gate, decay, and clustering.
- **Autonomy policy.** Four trust levels, allow/deny/queue rules, an irreversible-action gate, and taint tracking for untrusted content.
- **Observability and self-optimization.** Per-task cost, token, and outcome analytics, plus a propose-only optimizer that suggests routing and prompt changes.
- **Identity, leases, lifecycle.** Optional RBAC auth, one-orchestrator-per-repo leases with epoch fencing, and snapshot/restore/export.
- **Mission Control.** A web UI (React + shadcn/ui) for chat, the live event feed, memory, analytics, approvals, and system status.

## Requirements

- Node 22 or newer (Lunaris uses the built-in `node:sqlite`, `fetch`, and `node:test`)
- pnpm 9 or newer

## Installation

Lunaris is a pnpm + TypeScript monorepo. Clone it, install, and build:

```sh
git clone git@github.com:snag-9677/Lunaris.git
cd Lunaris
pnpm install
pnpm build
```

That is the whole installation. Two optional checks confirm everything is wired up:

```sh
pnpm test                # unit tests across all packages
node scripts/smoke.mjs   # offline end-to-end run using the mock provider
```

The first `pnpm install` may print a one-line warning about a missing `lunarisd` binary. It is harmless â€” the binary is generated by `pnpm build`, which you run next.

### Making the commands available

The CLI is `lunaris` and the daemon is `lunarisd`. They are not published to npm yet, so expose them with a shell alias (works as soon as the build finishes):

```sh
echo "alias lunaris='node $PWD/packages/cli/dist/main.js'" >> ~/.zshrc
echo "alias lunarisd='node $PWD/packages/daemon/dist/main.js'" >> ~/.zshrc
source ~/.zshrc
lunaris --help
```

If you prefer a real binary on your `PATH`, run `pnpm --filter @lunaris/cli link --global` instead. Either way, the rest of this guide assumes `lunaris` is callable.

## Quickstart

### 1. Initialize a project

```sh
cd ~/your-repo
lunaris init --name your-repo
```

This writes `lunaris.toml`, a `.aienv.sample`, and a `.lunaris/` state directory. The default model is `mock/echo`, which runs offline with no API key, so you can try the loop right away:

```sh
lunaris chat "create hello.txt that says hi, then summarize what you did"
lunaris status
lunaris events --tail 20
```

### 2. Connect a real model

Set the default model in `lunaris.toml`, then copy the env template and fill in your keys:

```sh
cp .aienv.sample .aienv
# ANTHROPIC_API_KEY=sk-ant-...
# DEEPSEEK_API_KEY=...
# OLLAMA_BASE_URL=http://localhost:11434
```

Lunaris loads `.aienv` per project and merges it into the environment without overwriting variables you have already exported, so shell exports and CI-injected secrets still take precedence. Keys never touch `lunaris.toml`, the logs, or the event log.

```sh
lunaris chat "add a /health endpoint and a test for it"
```

### 3. Run the daemon and Mission Control

```sh
lunarisd                       # HTTP + WebSocket on 127.0.0.1:7340, loopback only
```

The daemon serves the built UI at `http://127.0.0.1:7340`. During UI development you can instead run `pnpm --filter @lunaris/ui dev`, which proxies the API to the daemon.

Register a repository with the daemon and then drive it from the chat view or the API:

```sh
curl -X POST 127.0.0.1:7340/api/projects \
  -H 'content-type: application/json' \
  -d '{"root":"/absolute/path/to/your-repo"}'
```

## Configuration

### `lunaris.toml`

The project manifest is committed to the repository. It references API keys by environment-variable name and never stores the keys themselves.

```toml
[project]
id = "0192..."                            # stable lineage id, minted by `lunaris init`
name = "your-repo"

[models]
default = "anthropic/claude-sonnet-4-6"   # "<provider>/<model>"

[models.roles]                            # optional per-role overrides
coder = "anthropic/claude-sonnet-4-6"
reviewer = "deepseek/deepseek-chat"
researcher = "ollama/qwen3:8b"

[providers.anthropic]
keyEnv = "ANTHROPIC_API_KEY"

[providers.deepseek]
baseUrl = "https://api.deepseek.com"
keyEnv = "DEEPSEEK_API_KEY"

[providers.openai]
keyEnv = "OPENAI_API_KEY"

[providers.ollama]                        # local; base URL via OLLAMA_BASE_URL in .aienv

[budgets]
perCallUsd = 0.5
perDayUsd = 10.0
```

Supported providers are Anthropic, OpenAI, DeepSeek (and any OpenAI-compatible endpoint via `baseUrl`), local Ollama, and the `mock/echo` provider for offline runs. Budgets are reserved when a call is admitted and settled when it completes, so concurrent subagents cannot collectively exceed a cap.

### Autonomy policy

An optional `.lunaris/policy.yaml` controls what agents may do without asking. There are four levels: `0` read-only, `1` supervised, `2` autonomous within the workspace (the default), and `3` full-auto. Rules evaluate to `allow`, `deny`, or `queue`. Irreversible actions such as `git push`, deploys, package publishes, or `rm -rf` outside the repository always queue for approval regardless of level, and untrusted ("tainted") content tightens the active profile automatically.

```yaml
level: 2
tightenWhenTainted: true
allowlistedHosts: ["api.github.com"]
rules:
  - effect: deny
    tools: [run_bash]
    commands: ["curl * | sh"]
```

Queued actions are resolved with `lunaris approvals` or the approvals view in the UI.

### Environment variables

| Variable | Purpose | Default |
|---|---|---|
| `LUNARISD_PORT` | Daemon port | `7340` |
| `LUNARIS_AUTH` | Set to `on` to enable auth and RBAC on the daemon API | `off` (single loopback owner) |
| `LUNARIS_WEBHOOK_SECRET` | HMAC secret for `/hooks/:project/:source` | unset (loopback-only without it) |
| `LUNARIS_WEBHOOK_SECRET_<PROJECTID>` | Per-project webhook secret override | unset |
| `ANTHROPIC_API_KEY`, `DEEPSEEK_API_KEY`, `OPENAI_API_KEY` | Provider keys (whatever names your `keyEnv` settings use); set in `.aienv` or the shell | unset |
| `OLLAMA_BASE_URL` | Ollama endpoint; set in `.aienv`, overrides `lunaris.toml` | `http://localhost:11434` |

## HTTP API

The daemon listens on `127.0.0.1:7340` and refuses to bind to any non-loopback address. Remote access is expected to go through an SSH or Tailscale tunnel.

| Scope | Endpoints |
|---|---|
| Status | `GET /api/status`, `GET /api/version`, `GET /api/whoami`, `GET /api/projects` |
| Per project (`/api/projects/:id/â€¦`) | `POST goals`, `GET events`, `GET memory`, `GET memory/graph`, `GET analytics`, `GET\|PUT policy`, `GET approvals`, `POST optimize`, `GET proposals`, `GET\|POST queue`, `GET\|POST\|DELETE schedules`, `GET\|POST triggers`, `GET\|POST plugins` (+ `plugins/:pluginId/enable\|disable`), `POST snapshot`, `GET snapshots`, `POST restore`, `POST export` |
| Queues | `POST /api/approvals/:ticketId/resolve`, `POST /api/proposals/:proposalId/resolve` |
| Webhooks | `POST /hooks/:projectId/:source` (HMAC-verified) |

With `LUNARIS_AUTH=off` (the default) a single implicit loopback owner is permitted everything. With auth on, `POST /api/login` returns a bearer token, `POST /api/ws-ticket` mints a short-lived single-use ticket for the `WS /api/ws` upgrade, and RBAC capability checks apply to each route.

## Architecture

Lunaris is split into focused packages:

| Package | Responsibility |
|---|---|
| `@lunaris/core` | Shared contracts, the `lunaris.toml` manifest, UUIDv7 ids, the SQLite event store, analytics, schema migration, and `.aienv` loading |
| `@lunaris/gateway` | Multi-provider model gateway with streaming, tool-call, and vision normalization, plus the transactional budget ledger |
| `@lunaris/orchestrator` | The agent loop: roles, confined tools, subagent spawning with token attenuation, lease fencing, and the run journal |
| `@lunaris/memory` | Graph memory with a retention gate, similarity search, community clustering, and decay |
| `@lunaris/policy` | The policy decision point, autonomy levels, the irreversible-action class, taint tracking, and the approval queue |
| `@lunaris/optimizer` | Outcome ledger, success statistics, a routing bandit, and propose-only configuration changes |
| `@lunaris/plugd` | Plugin host for contributed tools and MCP server definitions |
| `@lunaris/scheduler` | Goal queue, cron parser, schedules, trigger rules, and the dispatcher |
| `@lunaris/identity` | Principals, password auth and sessions, the RBAC matrix, capability tokens, and the lease store |
| `@lunaris/lifecycle` | Snapshot, restore, bundle export/import, and two-level project identity |
| `@lunaris/daemon` | `lunarisd`: the HTTP and WebSocket API, project registry, goal runner, and scheduler loop |
| `@lunaris/cli` | The `lunaris` command |
| `@lunaris/ui` | Mission Control, built with React and shadcn/ui |

### Project layout

```
lunaris.toml          committed manifest
.aienv.sample         committed template for provider keys
.aienv                gitignored â€” your actual keys
.lunaris/
  state/              events.db, instance.json (machine-local id), queues
  memory/graph.db     knowledge-graph memory
  journal/            per-run transcript journals
  policy.yaml         optional autonomy policy
  plugins/            installed plugins
  snapshots/          snapshot archives
```

Machine-global state â€” the project registry, identity and lease databases, the agent-token signing key, and the node id â€” lives under `~/.lunaris/`.

## Moving between machines

Everything under `.lunaris/state` and `.lunaris/memory` is gitignored, so a fresh clone of a project starts with an empty AI environment. Hydrate it with:

```sh
lunaris adopt
```

This mints a fresh machine-local instance id and recreates the state directories. To carry a project's memory and history elsewhere, `lunaris export` produces a portable bundle (memory, proposals, and analytics, with secrets excluded); importing it always mints a new instance id so secret namespaces never collide.

## Status

All four phases described in the spec are implemented: the core loop and gateway, memory and policy and analytics, the optimizer and plugins and scheduler, and identity, leases, and lifecycle.

A few items from the spec are intentionally deferred past this version: the optimizer's golden-task A/B evaluation harness, plugin UI panels and sandboxing, cross-machine state sync and team memory, an external Postgres control plane for fleet mode, and self-updating release slots.
